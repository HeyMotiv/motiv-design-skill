This is a checkpoint. Don't proceed until the map and classifications feel right.

What you need: The approved plan with assignments.

Your task: Design the Human-Critical and Human-Preferred screens. This is where your creative vision lives.

  • Move to üü° Human-Preferred screens (these establish patterns)
  • Work in Figma, Sketch, code‚Äîwhatever you prefer

  • Think about what patterns you're establishing for AI to follow

What you need: Screenshots or HTML exports of your designed screens.

Run: /design-system

What to expect: AI analyzes your designs and extracts tokens, components, and patterns into a formal system.

  • Review extracted colors and their semantic names
  • Verify typography scale matches your intent
  • Check spacing tokens for accuracy
  • Review proposed motion specs (adjust if needed)
  • Run /design-system test to see the system applied
üîç

5

What you need: The extracted design system.

Run: /product-design generate

What to expect: AI generates all üü¢ AI-Extended screens using your design system. This is where the efficiency gain happens.

  • Review generated screens in a browser
  • Check that design system tokens were applied correctly
  • Verify emotional intent comes through
  • Request regeneration for screens that miss the mark

This is the quality gate. Don't ship until you're satisfied.

  • Review every screen, especially edge cases
  • Check transitions between screens make sense
  • Validate accessibility requirements are met
  • Make manual tweaks where AI got it 90% right
  • Promote screens to Human-Preferred if they need more work

  • Share with engineering lead for questions
  • Update as needed during implementation
Multi-Designer Teams

Coordinating Multiple Designers

When more than one designer is working on key screens, coordination becomes critical. Here's how to stay aligned.

1. Assign Ownership Clearly

Each Human-Critical screen should have one owner. Don't split ownership of closely related screens (e.g., onboarding steps 1-3 should all be one person). Use the screen inventory to track assignments.

2. Design Pattern-Setting Screens First

Before everyone dives into their assigned screens, identify which screens will set patterns for others. Design those first as a team, agree on the patterns, then let individuals apply them.

3. Extract Design System Early

Once you have 3-5 pattern-setting screens designed, run the design system extraction. This gives everyone a shared foundation to work from. Update as more screens are completed.

4. Run Deviation Reports Regularly

Use /design-system deviation-report after each batch of designs. This catches drift before it compounds. Address deviations as a team‚Äîdecide whether to normalize or update the system.

5. Sync on Emotional Intent

Different designers may interpret "confident and ready to act" differently. Sync on specific reference screens that exemplify the desired emotional tone. Use these as your north star.

The goal: Users shouldn't be able to tell which designer worked on which screen. The app should feel like it came from one unified creative vision. Deviation reports make this achievable at scale.

When Things Diverge

How Conflicts Get Resolved

When multiple designers contribute, inconsistencies are inevitable. The workflow has built-in mechanisms to catch and resolve them.

üé®

Scenario: Different Spacing Values

Designer A uses 20px card padding. Designer B uses 24px. Both are valid choices, but the app looks inconsistent.

‚úÖ

Resolution: The deviation report catches this automatically. You decide: normalize to 24px (design system value), or update the system to allow both with semantic meaning (compact vs. comfortable).

üî§

Scenario: Typography Drift

Over time, designers start using 18px for card titles instead of 16px. It looks better on some screens, worse on others.

‚úÖ

Resolution: The deviation report shows all instances. You can normalize to the system value, or add a new semantic token (title-large: 18px) and update the system. All decisions are tracked.

üåà

Scenario: Color Inconsistency

Designer A samples a blue from the Figma file, getting #2563EB. Designer B eyedrops from a compressed screenshot, getting #3B82F6. Both think they're using "primary blue."

‚úÖ

Resolution: The deviation report flags colors that are close but not exact matches. You normalize to the design system token, ensuring everyone uses the same source of truth.

üß©

Scenario: Different Component Variants

Designer A creates a card with icon on the left. Designer B creates a similar card with icon on top. Both make sense for their screens.

‚úÖ

Resolution: This might be valid variation! Review together‚Äîif both make sense, add them as documented variants (card-horizontal, card-vertical). If one is clearly better, normalize.

The key: Conflicts are surfaced early and resolved systematically. Instead of discovering inconsistencies in code review (expensive to fix), they're caught during design (cheap to fix). The deviation report isn't punitive‚Äîit's a tool for alignment.

Common Questions

FAQ

Answers to questions that come up frequently.

What if I want to design a screen that AI classified as AI-Extended?

‚ñº

Do it! The classifications are recommendations, not rules. If you feel strongly about a screen, move it to Human-Preferred and design it yourself. Trust your instincts‚Äîif you think it needs a human touch, it probably does. The system is here to serve you, not constrain you.

How do I know if my designs are "pattern-setting" enough for AI to follow?

‚ñº

Run /design-system test after extracting. If the test screens feel like they came from the same app as your human-designed screens, you're good. If they feel generic or off, your pattern-setting screens might be too inconsistent or not distinctive enough. Add more variety to your keyframes and re-extract.

Can I use Figma instead of HTML for my designs?

‚ñº

Yes! Export your Figma frames as screenshots and use those for design system extraction. The AI analyzes the visual output, not the source format. Many teams find it faster to work in Figma for exploration, then export for extraction.

What if AI generates something I don't like?

‚ñº

You have options: (1) Request regeneration with more specific instructions, (2) Manually tweak the output, (3) Promote the screen to Human-Preferred and design it yourself, or (4) Adjust the design system tokens and regenerate. The workflow is iterative‚Äînothing is permanent until you ship.

How does this work with existing design systems?

‚ñº

If you already have a design system, you can import it instead of extracting from screens. Use /design-system import with your existing tokens. The workflow will use your system as the source of truth for AI generation.

Does this work for web apps or just mobile?

‚ñº

Both! The workflow is platform-agnostic. For web apps, you may have more breakpoints to consider, but the principles are the same. The design system extraction handles responsive patterns, and AI generation can produce screens at multiple viewport sizes.

How long does the full workflow take?

‚ñº

It depends on the scope, but the workflow is designed to front-load effort on the screens that matter most. Teams typically find they get to a reviewable full-app prototype faster than traditional approaches, with higher consistency across screens.

What happens when requirements change mid-project?

‚ñº

Update the discovery doc, re-run the plan step to update the screen inventory, and regenerate affected screens. Because AI-generated screens are derived from the design system, changes propagate automatically. Human-designed screens may need manual updates, but there are fewer of them.
